## Background

`video-service/internal/worker/vectorworker` currently uses a two-stage ASR strategy in `hierarchical` mode:

1. Coarse segmentation ASR: split the full video into coarse windows and transcribe each coarse window.
2. Refine segmentation ASR: after the LLM proposes fine-grained content segments, run per-segment ASR again to align boundaries and generate embedding input.

This second stage exists for a valid reason: it reduces hard cuts where a segment starts or ends in the middle of a sentence. That helps preserve semantic completeness for embeddings and improves any later segment text display.

The current implementation, however, multiplies ASR cost because refine-stage alignment does not perform one ASR call per segment. Instead, it probes multiple candidate boundaries and may also perform tail fallback probing. For a long video, this can push total ASR audio duration far beyond the original media duration.

Observed problem:

- Test video duration: about 6574s
- Total ASR consumption: about 13000s

The goal is to preserve the anti-hard-cut intent while significantly reducing ASR usage.

## Goals

- Reduce total ASR audio consumption in `hierarchical` mode.
- Preserve the quality guard that avoids hard-cutting sentences at fine-segment boundaries.
- Keep the change localized to the vector worker pipeline.
- Avoid database schema changes in the first iteration.

## Non-Goals

- Rewriting the coarse segmentation pipeline.
- Changing LLM segmentation prompt semantics.
- Replacing the ASR provider or embedding model.
- Introducing a new persistence model for coarse transcripts in this iteration.

## Root Cause Summary

The excess ASR usage comes from the refine phase in `internal/worker/vectorworker/tasks/asr.go`.

Current behavior:

- Coarse ASR already provides `coarseItems[start_sec, end_sec, text]`.
- Fine segments are generated by the LLM.
- `RefineSegmentsASRAndEmbed` loads unfinished fine segments and, for each segment, calls `alignSegmentForRefine`.
- `alignSegmentForRefine` probes multiple boundary candidates using `probeRange`.
- Each `probeRange` call extracts audio and sends a full ASR request.
- Tail fallback can trigger more ASR requests for the same segment.

This means a single fine segment may trigger several ASR calls before embedding input is finalized.

## Chosen Approach

Use a balanced strategy:

- Default to reusing coarse ASR text for fine-segment embedding input.
- Only segments with `boundary_confidence == "low"` may trigger refine-stage ASR.
- For those low-confidence segments, run at most one targeted refine ASR call.
- That targeted refine ASR should use a small time expansion around the segment boundary to preserve the original anti-hard-cut intent.

This approach retains the quality guard where the pipeline itself has already signaled boundary uncertainty, while removing repeated ASR probing from the common path.

## High-Level Design

### 1. Keep coarse ASR as the canonical default transcript source

The coarse stage remains unchanged and continues producing ordered `coarseItems` with:

- `start_sec`
- `end_sec`
- `text`

These coarse items are already available during hierarchical segmentation before fine segments are inserted.

### 2. Build fine-segment embedding input from coarse transcript slices

After LLM fine segments are normalized, each fine segment will first derive its transcript input from overlapping coarse ASR text instead of immediately triggering segment-level ASR.

The derived embedding input format remains consistent with current intent:

- `content_summary`
- plus transcript text derived for that segment

This keeps title-style segment semantics while still including actual spoken content.

### 3. Restrict refine ASR fallback to low-confidence segments only

Only fine segments marked with `boundary_confidence == "low"` should bypass the coarse-text-only path.

Reasoning:

- The LLM already emits boundary confidence.
- Low-confidence boundaries are the most likely to cut through incomplete spoken sentences.
- Restricting fallback to these segments minimizes ASR usage while preserving the original boundary-quality protection where it matters most.

### 4. Replace multi-probe refine ASR with single-shot expanded-window ASR

For a low-confidence segment:

- Do not use multi-candidate boundary probing.
- Do not use iterative tail extension probing.
- Instead, create one expanded time window around the fine segment.
- Run one ASR request on that expanded audio.
- Use the returned text as the segment transcript input.

Recommended expansion behavior for initial implementation:

- Expand start backward by a small fixed number of seconds.
- Expand end forward by a small fixed number of seconds.
- Clamp to video bounds.
- If there is a next segment start, the expanded end must not cross that next start.

This preserves the anti-hard-cut purpose without turning boundary repair into repeated ASR exploration.

## Detailed Behavior

### Transcript selection rules

For each fine segment, choose transcript input by the following rules:

1. If `boundary_confidence != "low"`:
   - Build transcript text from overlapping coarse ASR items.
   - Do not run refine ASR.
2. If `boundary_confidence == "low"`:
   - Expand the segment time range slightly.
   - Run exactly one refine ASR request on the expanded audio.
   - Use that result as transcript text.
3. If refine ASR fails for a low-confidence segment:
   - Fall back to the coarse-text-derived transcript for that segment.
   - Do not retry multiple alternative windows inside the same task execution.

### Coarse-text-derived transcript construction

The implementation should add a helper that derives segment text from overlapping coarse items.

Expected behavior:

- Gather ordered coarse items that overlap the fine segment time range.
- Concatenate their normalized text in time order.
- Trim whitespace and collapse empty items.
- Return empty string only when no overlapping coarse text exists.

This helper is intentionally simple in the first iteration. It does not need word-level timestamp alignment.

### Expanded refine window behavior for low-confidence segments

The initial implementation should use a fixed, small expansion window.

Requirements:

- Expansion must be deterministic.
- Expansion must be bounded by `0` and `videoDurationSec`.
- Expansion must not overlap beyond the next segment's nominal start when the next segment exists.
- Segment duration passed into ASR must always remain positive.

The exact expansion constant can remain an implementation detail, but it should be small enough to cap cost and large enough to reduce sentence truncation risk.

### Embedding input composition

The final text passed to embedding should preserve the current strategy:

- start with `content_summary`
- append transcript text when present

If transcript text is empty:

- use `content_summary` alone

This keeps embeddings usable even when ASR fallback fails or coarse transcript coverage is sparse.

## Code Changes

### `internal/worker/vectorworker/task.go`

- Preserve existing coarse ASR execution.
- Pass the generated `coarseItems` into the refine-stage processing path instead of discarding them after LLM segmentation.
- Update the refine-stage call signature so it can derive transcript text from coarse ASR results.

### `internal/worker/vectorworker/tasks/asr.go`

- Refactor `RefineSegmentsASRAndEmbed` to accept coarse transcript context.
- Split transcript production into two paths:
  - coarse-text-derived path for normal segments
  - single-shot refine ASR fallback path for low-confidence segments
- Remove default reliance on `alignSegmentForRefine` for the common path.
- Ensure low-confidence fallback uses at most one ASR request per segment.
- If practical, keep existing alignment helpers available for now only if still needed by tests or compatibility, but they should no longer drive the primary runtime path.

### New helper logic in `tasks`

Add small focused helpers rather than a large abstraction layer:

- helper to derive transcript text from coarse items for a fine segment
- helper to compute the expanded ASR window for low-confidence segments
- helper to decide whether a segment requires refine fallback based on `boundary_confidence`

The code should stay local to the current task package and avoid introducing broader cross-package APIs.

## Error Handling

- If coarse-text derivation returns empty text for a non-low-confidence segment, continue with `content_summary` only.
- If low-confidence refine ASR fails, log the fallback and continue with coarse-text-derived content instead of failing the whole task immediately for that segment.
- If embedding fails, retain the current task-level failure behavior.

This choice intentionally treats low-confidence refine ASR as a quality enhancement, not a hard dependency.

## Logging and Observability

Add or update logs so ASR cost behavior can be validated after rollout.

Recommended counters or debug logs:

- number of fine segments using coarse-text-only path
- number of low-confidence segments using refine fallback
- number of fallback ASR failures that degraded to coarse text
- total low-confidence fallback segment duration submitted to ASR

This will make it easy to compare before and after behavior on long videos.

## Testing Strategy

Tests should be added before production changes and should verify behavior, not just implementation details.

Required coverage:

- fine segment with non-low confidence uses coarse transcript path and does not invoke ASR
- fine segment with `boundary_confidence == "low"` invokes refine ASR once
- refine fallback window is clamped within video duration and next segment boundary
- refine ASR failure on a low-confidence segment falls back to coarse text instead of triggering repeated ASR attempts
- embedding input still combines `content_summary` and transcript text

Prefer unit tests in `internal/worker/vectorworker/tasks` with stubs for ASR and FFmpeg behavior where needed.

## Verification Plan

After implementation, validate on the same or similar long video used in the original report.

Success indicators:

- total ASR audio duration drops substantially from the current baseline
- low-confidence segments remain readable and not obviously hard-cut
- embedding generation still completes successfully
- no regression in segment persistence or task completion behavior

## Risks

### Risk: coarse transcript granularity is too rough

Because coarse transcript text is reused without word-level alignment, some non-low-confidence segments may contain slightly broader text than their exact spoken range.

Why acceptable in v1:

- embeddings care more about semantic coverage than exact subtitle boundaries
- low-confidence segments still retain ASR protection

### Risk: low-confidence fallback may still include extra neighboring text

The expanded single-shot ASR window may capture slightly more surrounding context than the exact ideal sentence boundary.

Why acceptable in v1:

- this is preferable to sentence truncation
- it is still dramatically cheaper than multi-probe ASR

### Risk: low-confidence fallback failure weakens text quality

If a fallback ASR call fails, the segment will degrade to coarse text instead of failing the whole task.

Why acceptable in v1:

- preserves pipeline completion
- avoids retry storms
- still produces usable embeddings in most cases

## Rollout Recommendation

Implement this as the first optimization step without schema changes.

If more reduction is needed later, a second iteration can consider:

- persisting coarse ASR text for reuse across retries
- more precise transcript slicing rules
- configurable expansion seconds for low-confidence fallback

The first iteration should stay minimal and focused on removing repeated ASR probing from the common path.
