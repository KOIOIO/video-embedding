# 智能教学视频分析与推荐系统讲解型 PPT 设计文档

## 背景

当前仓库是多项目容器，但后续实际部署和对外提供能力的目标服务是 `video-service/`。本次 PPT 需要围绕这个 Go HTTP 视频服务展开，面向混合受众进行讲解，包括：

1. 技术团队 / 架构评审
2. 业务方 / 管理层
3. 客户 / 甲方汇报

用户明确要求内容不能只讲优点，必须同时呈现设计亮点、处理优势、工程权衡、当前限制和后续演进方向。

## 目标

本次 PPT 的主目标是解释系统全貌，方便多方对齐；同时兼顾以下次目标：

1. 让听众认可当前方案的工程合理性。
2. 为后续资源投入、优化或演进提供论据。
3. 客观暴露问题，而不是形成纯宣传稿。

## 非目标

1. 不把历史工程 `nlp-video-project/` 作为主讲对象。
2. 不把 `hls-web/` 前端调试工程作为主线。
3. 不把 PPT 做成逐行代码讲解。
4. 不把系统包装成“没有缺点的成熟平台”。

## 推荐方案

采用“分层递进型”讲解结构。原因如下：

1. 适合混合受众，前半段业务能听懂，后半段技术站得住。
2. 可以先建立系统全貌，再逐步展开关键链路和设计思想。
3. 便于单独安排“问题与限制”章节，避免优点与风险混讲后重点失焦。

不采用其他方案的原因：

1. 按链路拆解型对技术听众更顺，但业务和客户在中段容易失去整体视角。
2. 对比评审型更适合架构评审，不适合本次面向多方的统一汇报。

## 核心信息定位

PPT 需要把该系统定义为：

> 一个面向 Java 业务系统的独立视频 HTTP 后端服务，不只承担上传和播放，还覆盖视频处理、异步转码、对象存储代理、题目关联推荐、观看回流和向量化扩展能力。

同时需要明确以下判断：

> 当前架构是务实可部署、能力较完整、具备演进空间的中等复杂度系统；但它并不是零成本扩展的平台化方案，仍然存在外部依赖重、异步链路复杂、资源竞争明显、运维治理要求较高等现实约束。

## 受众表达策略

### 面向业务方 / 管理层

重点回答：

1. 这个系统解决了什么问题。
2. 为什么值得独立出来做成服务。
3. 它相比简单视频上传接口多了哪些业务价值。
4. 当前继续投入的价值在哪里。

表达原则：

1. 少用术语堆砌。
2. 用“同步接口 + 异步处理 + 推荐增强”作为主线。
3. 风险讲清楚，但不夸大成“不能上线”。

### 面向技术团队 / 架构评审

重点回答：

1. 系统边界如何划分。
2. 同步链路和异步链路如何配合。
3. Redis Streams、Worker、S3、PostgreSQL、pgvector、AI provider 分别承担什么职责。
4. 当前设计的优点和瓶颈分别在哪里。

表达原则：

1. 架构图和链路图必须能自洽。
2. 既讲设计亮点，也讲权衡代价。
3. 不回避统一 worker、共享资源竞争、AI 外部依赖等问题。

### 面向客户 / 甲方

重点回答：

1. 这套系统已经具备哪些能力。
2. 在什么场景下它是合适的。
3. 哪些场景下它还需要进一步建设。
4. 为什么说它不是只会“存视频”的简单后端。

表达原则：

1. 强调完整链路和业务闭环。
2. 避免过度承诺极限性能和超大规模能力。
3. 主动说明适用与不适用范围，建立可信度。

## 建议页结构

建议页数控制在 14 页左右，结构如下。

### 1. 封面

- 标题：智能教学视频分析与推荐系统系统架构与工程设计
- 副标题：面向 Java 业务系统的视频处理、推荐与播放服务
- 风格：正式、可信、可同时用于客户介绍和技术评审

### 2. 执行摘要

目的：用一页让所有受众建立共识。

必须表达：

1. 这不是单纯的视频上传接口。
2. 这是一个集视频处理、推荐、播放代理、观看回流于一体的综合后端。
3. 当前方案实用且可部署，但有明确边界和后续治理要求。

### 3. 项目背景与建设目标

重点说明：

1. 为什么从 Java 主系统中独立出视频能力。
2. 为什么要引入异步处理链路。
3. 为什么推荐、播放和观看回流要在一个服务中闭环。

### 4. 系统边界

需要明确列出“做什么”和“不做什么”。

做：

1. 视频上传
2. HLS 转码与封面处理
3. 视频列表、删除、发布、推荐状态维护
4. 题目关联推荐
5. 观看记录上报
6. 题库查询
7. 对象存储视频资源代理访问

不做：

1. 超大规模 CDN 分发平台
2. 低延迟实时直播处理平台
3. 通用 AI 平台
4. 复杂多租户媒体编排系统

### 5. 总体架构图

主体元素：

1. Java 业务系统
2. 浏览器 / 播放器
3. `cmd/httpapi`
4. `videoapp.Service`
5. Redis Streams
6. PostgreSQL + pgvector
7. S3 / RustFS
8. `cmd/worker`
9. transcode worker / vector worker
10. FFmpeg
11. ASR / Embedding / LLM

核心口径：

1. 对外是统一 HTTP 服务。
2. 对内是同步 API 与异步任务处理组合。
3. 对象存储负责文件，数据库负责元数据与检索能力。

### 6. 核心业务链路图

建议拆成 4 条主链路：

1. 上传与转码链路
2. 推荐与向量化链路
3. 播放与资源代理链路
4. 观看记录回流链路

需要突出：

1. 同步请求和异步任务的边界。
2. 队列、状态回写、对象存储和外部 AI 的位置。
3. 多视频并发与单视频内部并发的差异。

### 7. 关键模块拆解

每个模块都要讲“职责、优点、代价”。

模块包括：

1. 接入层：Gin Router + REST API
2. 应用层：`videoapp.Service`
3. 异步层：transcode worker / vector worker
4. 存储层：PostgreSQL + pgvector、Redis Streams、S3
5. 外部能力层：FFmpeg、ASR、Embedding、LLM

### 8. 设计思想与架构原则

必须覆盖以下原则：

1. 服务解耦：Java 主系统通过 HTTP 调用独立视频服务。
2. 异步优先：长耗时链路不阻塞主请求。
3. 存算分离：对象存储承载媒体文件，数据库承载元数据和检索。
4. 降级优先：AI 不可用时优先返回可用结果。
5. 渐进演进：先构建可运行系统，再逐步推进阶段化和分布式化。

### 9. 设计亮点

需要突出但不能吹过头：

1. HTTP 服务与异步 worker 解耦。
2. Redis Streams + 消费者组 + 死信流具备基础可靠性能力。
3. 推荐链路具备 AI 降级与补偿思路。
4. 向量化链路已经为阶段化拆分预留结构。
5. 对外 REST 接口统一，便于 Java 对接和 Swagger 协作。

### 10. 处理优势与工程收益

重点讲结果，而不是只讲技术名词：

1. 用户请求不需要长期阻塞在转码和向量化任务上。
2. 多视频处理具备并发能力，整体吞吐更合理。
3. 对象存储降低本地磁盘耦合。
4. 题目推荐与视频服务打通后形成业务闭环。
5. 统一 HTTP 服务降低 Java 系统接入复杂度。

### 11. 风险、短板与当前限制

这是本次 PPT 的必备章节，必须单独成页。

需要明确指出：

1. HTTP 服务启动依赖 PostgreSQL、Redis、对象存储与数据库扩展，基础设施耦合较重。
2. 统一 worker 入口虽然简单，但转码和向量化共享资源，容易互相影响。
3. AI 服务和 FFmpeg 都是重依赖，上游波动会传导到处理稳定性和时延。
4. 视频级并发叠加单视频内部并发，会把瓶颈集中到 FFmpeg、对象存储、数据库和 AI 服务。
5. 推荐链路虽然可降级，但降级意味着结果质量可能下降。
6. 当前方案更适合中等规模，若继续放大规模需要进一步拆分与治理。

表达要求：

1. 风险要客观，不渲染恐慌。
2. 问题要和后续建议对应，避免只批评不收束。

### 12. 运维与治理挑战

重点讲：

1. 故障面多：数据库、缓存、对象存储、AI provider、FFmpeg 都可能成为问题源。
2. 任务堆积、死信增长、AI 限流、对象存储延迟都需要监控。
3. 真正稳定运行不仅靠代码，还依赖容量治理、监控告警和故障处理机制。

### 13. 适用场景与不适用场景

适用：

1. 教育视频平台
2. 中等规模内容服务
3. 需要题目关联推荐或内容理解能力的场景

不适用：

1. 超大规模直播分发
2. 极低延迟实时媒体处理
3. 强多租户隔离的复杂平台化场景

### 14. 演进方向与总结

建议合并收束：

1. worker 拆分部署
2. 向量化链路进一步阶段化 / 分布式化
3. 加强任务编排、监控和容量治理
4. 优化推荐质量评估与缓存策略
5. 继续收敛兼容接口和历史路径

结论口径：

> 这是一个工程上务实、链路完整、可落地的视频服务；它已经能支撑当前业务闭环，但若要持续支撑更大规模或更复杂场景，后续关键在于异步链路拆分、资源治理和运维体系完善。
