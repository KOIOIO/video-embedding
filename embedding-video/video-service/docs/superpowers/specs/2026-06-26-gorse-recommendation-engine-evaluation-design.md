# Gorse Full-Chain Recommendation Engine Design

## Goal

Replace the current custom Python two-tower training, embedding artifact publishing, PostgreSQL vector recall, and personalized video recommendation serving path with Gorse-managed recommendation training and serving.

The Go service remains the product API and source-of-truth domain system. Gorse becomes the recommendation engine.

## Decision

Use Gorse as the primary recommendation engine for personalized video segment recommendation.

Gorse should own:

1. User, item, and feedback data model for recommendation.
2. Recommendation model training.
3. Offline recommendation cache generation.
4. Online recommendation retrieval and ranking.
5. Read/positive/negative feedback learning loop.

The project should stop treating `two-tower-training/` as the production recommendation path after Gorse is validated. The Python trainer, embedding importer, `edu_user_tower_embedding`, and `edu_video_item_embedding` should become rollback-only during the transition and removable after a stable rollout.

This is viable because Gorse models users, items, and feedback directly; supports user/item labels and item embedding vectors; supports explicit negative feedback; trains collaborative filtering and ranking models; generates cached recommendations; exposes REST APIs; and has master/worker/server roles for model training and serving. Official docs describe users/items/feedback as the core data collections, feedback type configuration, negative feedback behavior, retrieval/ranking pipeline, matrix-factorization collaborative filtering, and REST recommendation APIs.

## What “Full Chain” Means Here

Full chain means:

```text
local domain data
  -> sync into Gorse users/items/feedback
  -> Gorse trains and refreshes recommenders
  -> Go calls Gorse recommendation APIs
  -> Go hydrates returned segment IDs
  -> Go records exposure/watch/reaction
  -> feedback sync pushes those events back into Gorse
```

Full chain does not mean Gorse replaces the HTTP API, local video tables, object storage, HLS playback, vector worker, or content embedding generation. Those remain product/domain responsibilities.

## Current System Being Replaced

The current personalized recommendation path is:

1. Export samples from PostgreSQL into CSV.
2. Train Python/PyTorch two-tower model.
3. Write embedding artifacts.
4. Gate offline metrics.
5. Import user/item embeddings into PostgreSQL `vector(64)` tables.
6. Publish active model version.
7. Serve `random-play` by reading user tower embedding and querying item embedding vectors.
8. Rebuild individual user tower embeddings after reactions.

This has produced unstable offline metrics because current data is small: around 7 users, 55 items, and 380 aggregated samples. TopK metrics also become distorted when Top20 covers most of a user's remaining candidate set.

## Data Mapping

### Users

Map `sys_user` rows into Gorse users:

```text
UserId = sys_user.id as string
Labels = grade_id, class_id, user_type, mastery summary, recent subjects, recent knowledge ids
Comment = compact debug string
```

Labels should include learning-state fields already exported by `tools/export_two_tower_samples`, but as Gorse user labels rather than Python model features.

### Items

Map each recommendable `edu_video_segment` into a Gorse item:

```text
ItemId = edu_video_segment.id as string
IsHidden = true when segment/video is deleted, disabled, unpublished, unplayable, or administratively hidden
Categories = subject and knowledge tags
Timestamp = segment/video create or publish time
Labels = video_id, title, summary, knowledge tags, duration, counters, segment embedding vector when available
Comment = title + segment summary for dashboard inspection
```

Gorse item labels can carry embedding vectors generated by the current vector worker. That lets the recommendation engine use multimodal/text-derived content signals without maintaining a separate two-tower item embedding table.

### Feedback

Use explicit feedback types instead of synthetic binary labels:

```text
double_like       positive, value 3
like              positive, value 2
watch             read feedback, value = watch ratio or duration-normalized score
watch>=0.6        positive via Gorse positive feedback type condition
exposure          read feedback, value 1
dislike           negative feedback, value 1
```

Do not map exposure-no-click to Gorse negative feedback in the first version. Gorse negative feedback has strong semantics: explicitly disliked items are not recommended to that user again, and similar items are deprioritized. Only real `dislike` should be negative.

## Gorse Configuration Direction

Recommended first production-like configuration:

```toml
[recommend.data_source]
positive_feedback_types = ["like", "double_like", "watch>=0.6"]
read_feedback_types = ["exposure", "watch"]
negative_feedback_types = ["dislike"]
positive_feedback_ttl = 0
item_ttl = 0

[recommend]
cache_size = 100
cache_expire = "72h"

[recommend.collaborative]
type = "mf"
fit_period = "60m"
fit_epoch = 100
optimize_period = "360m"
optimize_trials = 10

[recommend.ranker]
type = "fm"
recommenders = [
  "latest",
  "popular",
  "collaborative",
  "item-to-item/neighbors",
  "user-to-user/neighbors"
]
```

Use Gorse's LLM ranking only after the collaborative/ranking baseline is stable. LLM ranking is not part of the first migration because it adds cost, latency, and provider failure modes.

## API Ownership

The frontend continues to call the Go service.

Go service responsibilities:

1. Call Gorse for recommendation IDs.
2. Hydrate IDs into existing response DTOs.
3. Enforce video availability, status, deletion, permission, and playback rules.
4. Record `edu_recommend_exposure`, watch records, and reactions.
5. Push new feedback to Gorse either synchronously with timeout or asynchronously through a sync worker.
6. Fall back when Gorse is down or sparse.

Gorse responsibilities:

1. Store recommendation users/items/feedback.
2. Train models.
3. Generate recommendation candidates.
4. Rank candidates.
5. Apply read/negative feedback filtering.

## Endpoint Mapping

### `GET /api/video-segments/random-play`

Primary Gorse path:

```text
GET /api/recommend/{user_id}?n={candidate_limit}
```

Then Go hydrates returned `ItemId` values as `video_segment_id` and returns existing response shape.

### `GET /api/recommendations` / list-style personalized recommendation

Use the same Gorse recommendation path, then hydrate and persist local recommendation records.

### `POST /api/recommendations/by-question`

This endpoint currently means semantic question-to-video matching, not pure personalized recommendation. There are two acceptable options:

1. Keep it as semantic search and mark it outside the Gorse replacement scope.
2. Use Gorse as a post-filter/ranker after local semantic retrieval if the product wants personalized ordering for question-matched candidates.

Do not force pure Gorse recommendations into `by-question` unless product requirements change from “match this question” to “personalize video suggestions for this user”.

## Rollout

### Phase 0: Local Feasibility

- Add Gorse to local docker-compose.
- Sync all current users/items/feedback.
- Verify active users get recommendations.
- Verify disliked segments are not returned to the same user.
- Verify hidden/deleted/unplayable items are not returned after sync.

### Phase 1: Shadow Full Chain

- Gorse trains and serves recommendations.
- Go still returns existing two-tower/random results.
- Go records Gorse candidate IDs and current returned IDs for comparison.
- Compare candidate validity, coverage, overlap, watch/click conversion, and dislike hits.

### Phase 2: Gorse Primary With Fallback

- `random-play` and personalized list endpoints use Gorse first.
- Existing two-tower/random path remains fallback.
- Python scheduled trainer is disabled but not deleted.

### Phase 3: Retire Python Two-Tower

- Stop `two_tower_trainer` permanently.
- Stop importing two-tower embeddings.
- Keep old embedding tables read-only for one rollback window.
- Remove Python training service and importer after stable production metrics.

## Success Criteria

Gorse can fully replace the Python two-tower chain when:

1. It returns enough valid playable video segment IDs for active users.
2. Explicitly disliked segments are not returned to the same user.
3. Watch/click rate is equal to or better than current path.
4. Coverage does not collapse to a tiny popular subset.
5. Cold-start users receive acceptable latest/popular/content-label fallback recommendations.
6. Gorse outages do not break public APIs.
7. Operations can run through docker-compose or production orchestration without manual model artifact handling.

## Risks

Small data still limits personalization quality. Gorse reduces custom ML maintenance, but it cannot create preference signal that does not exist.

Gorse adds a separate stateful service with database/cache/blob state. We need backup, health checks, API keys, timeout handling, and fallback.

Item ID synchronization is now critical. The Go service must defensively hydrate only valid segment IDs.

Question-based semantic matching is not the same problem as personalized recommendation. Keep that boundary explicit.
