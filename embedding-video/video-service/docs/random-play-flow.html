随机播放推荐链路图

当前配置是后端先调用 Gorse，Gorse 内部优先 collaborative；如果 Gorse 需要 fallback，第一层会调用 external/two_tower。后端拿到候选片段 ID 后，还会做可播放、发布、推荐开关、已看、点踩和最近播放过滤。

当前推荐优先级
  1. Gorse collaborative
  2. Gorse fallback: external/two_tower
  3. 后端 knowledge_match
  4. 后端数据库随机片段
HTTP/API Gorse 双塔 external 后端兜底 数据输入/训练 Redis 最近去重

为什么不是纯随机

接口名字叫 random-play，但有 user_id 时先走推荐服务。只有 Gorse、知识点匹配都拿不到可用片段时，才会真正查库随机选一个。

双塔现在的位置

当前不是后端直接优先双塔，而是 Gorse fallback 里的 external/two_tower。Gorse 需要兜底时才会调用后端双塔接口。

最终谁保证可播放

模型只给候选片段 ID。后端 hydrate/filter 会保证视频已发布、可播放、未删除、未看过、未点踩，并记录曝光和最近返回。

推荐算法内部基础原理

随机播放实际是一条多级召回链路：前面的算法负责尽量给出更个性化的候选片段，后面的算法负责在数据不足、服务不可用或候选被过滤光时兜底。所有策略最终都只产出候选 segment_id，真正的业务可播放性由后端统一过滤和曝光记录保证。

Gorse collaborative：从群体行为里找相似偏好

输入：gorsesync 把用户、视频片段、观看、曝光、点赞、点踩等反馈同步进 Gorse，item_id 对应 video_segment_id。

原理：协同过滤不只看片段文本，而是看“哪些用户和哪些片段发生过类似互动”。如果两个用户的观看/点赞模式相近，系统会把其中一个用户喜欢但另一个还没看过的片段排到前面。

在链路中的作用：当前配置优先请求 Gorse，后端拿到 Gorse 返回的片段 ID 后，再按原顺序 hydrate/filter，并把曝光写回 Gorse 作为下一轮反馈。

external/two_tower：用户向量和片段向量做近邻匹配

输入：训练流水线把用户行为和片段内容训练成两套 embedding：user tower 表示用户当前兴趣，item tower 表示视频片段特征。

原理：在线推荐时取用户向量，在 item embedding 表里找距离最近的片段。距离越小，代表模型认为用户兴趣和片段内容越匹配，分数会换算成 1 / (1 + distance)。

在链路中的作用：它当前不是后端主入口，而是 Gorse 的 external fallback。Gorse 协同过滤数据不足或需要补候选时，才会调用这个后端接口返回片段 ID。

knowledge_match：用薄弱知识点召回相关片段

输入：后端读取用户掌握度较低的知识点，把知识点名称和描述拼成查询文本，再生成 embedding。

原理：用这个“薄弱知识画像向量”去匹配视频片段的内容向量，优先召回语义上能补弱项的片段；召回后还会用知识点词命中做一次轻量重排，标题和知识标签命中权重更高。

在链路中的作用：当 Gorse 没有可用结果时，后端会走这一层。它更偏教学补弱，不依赖协同行为，因此适合作为冷启动或稀疏用户的兜底。

数据库随机：最后兜底，也是高峰风险点

输入：业务库里已经发布、未删除、转码完成、片段可播放的视频片段集合。

原理：当前实现会在可播放池上执行 ORDER BY RANDOM() LIMIT 1；如果用户有最近返回记录，会先排除 Redis 最近去重集合里的片段，减少短时间重复播放。

风险：ORDER BY RANDOM() 需要给候选行生成随机值再排序。数据到几百万级、上游推荐全部失效且流量涌入时，数据库 IO 和 CPU 会被随机排序打满，兜底路径可能反过来触发雪崩。

定位：它只能作为低流量防止空结果的临时兜底，不应承担高峰期主保护。生产上更稳的做法是预热可播放片段池、用随机游标/随机 id seek、维护采样表或把兜底候选放进 Redis。

统一过滤和反馈：把模型候选变成可播结果

过滤：无论候选来自 Gorse、双塔还是知识点匹配，后端都会过滤未发布、不可播放、未转码完成、已看、点踩、关闭推荐和最近返回的片段。

回流：最终返回的片段会记录 recommendation exposure；播放、点赞、点踩等行为再回到 Postgres，并由 gorsesync 同步给 Gorse，形成下一轮训练和推荐依据。

为什么按这个顺序降级

Gorse 优先：它能综合群体互动、用户标签、物品标签和反馈，适合在线主推荐。

双塔补候选：它适合用训练好的向量做快速召回，但当前被放在 Gorse external fallback 中，避免两套在线主链路互相竞争。

知识点和随机兜底：前者保证教学相关性；后者只保证低流量下尽量不空返。越往后个性化越弱，数据库随机在大表高峰下还会牺牲稳定性，需要尽快替换成轻量采样兜底。
