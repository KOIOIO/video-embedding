# video-service 下游服务化改造评审文档

> 文档性质：服务化差距评审和建议清单，不是当前 API 契约或已完成事项清单。现行接口、配置和启动方式以 `../README.md`、`../configs/` 和 `../internal/http/router/router.go` 为准；本次按 2026-07-15 的代码状态复核。

## 1. 文档目的

本评审文档用于判断 `video-service/` 是否已经适合作为上游业务系统，尤其是 Java 系统，长期依赖的下游能力服务，并给出分优先级的改造建议。

这里讨论的重点不是“功能是否能跑通”，而是：

1. 是否适合作为标准下游服务被多个上游接入
2. 是否具备稳定的契约、鉴权、幂等、任务治理和流量保护能力
3. 哪些能力需要优先补齐，才能从“功能完整的视频后端”提升为“可治理的下游服务”

## 2. 当前项目定位

根据当前 README、代码结构和现有能力，`video-service/` 已经具备一个独立视频 HTTP 服务的核心能力，包括：

1. 视频上传
2. HLS 转码与封面生成
3. 视频列表、删除、发布、推荐状态维护
4. 基于题目的视频片段推荐
5. 观看记录上报
6. 题库查询
7. 对象存储视频资源代理访问
8. 转码 worker 与向量化 worker 的异步处理链路

从“能否作为一个后端服务独立运行”这个角度看，当前项目已经成立。

但从“能否作为一个成熟的下游能力服务长期承接多个上游系统”这个角度看，当前还存在明显差距。

## 3. 总体判断

当前 `video-service/` 更像：

- 一个功能完整的视频能力后端

而还不是：

- 一个成熟、标准化、可治理的下游服务产品

它目前的主要短板，不在单一业务功能，而在服务治理能力，包括：

1. 鉴权和授权边界不够清晰
2. 接口契约还偏项目内部风格，不够平台化
3. 异步任务状态治理还不够标准
4. 幂等和回调能力不足
5. 风暴保护和流量治理不够系统
6. 可观测性更偏开发排障，不够面向服务治理
7. 配置与并发模型对大规模上游接入不够保守

## 4. P0：必须优先补齐的能力

这些能力是“作为下游服务长期被依赖”的最低门槛。

### 4.1 鉴权与授权机制标准化

#### 当前问题

项目 README 已经明确 Java 会直接调用它，但代码和文档层面尚未形成清晰、稳定的统一鉴权模型。

对于下游服务而言，不能默认：

1. 谁都能上传
2. 谁都能播放
3. 谁都能删除
4. 谁都能查询任意任务状态

#### 建议

至少统一一种标准机制：

1. 服务间鉴权
   - JWT
   - HMAC 签名
   - 网关透传身份
2. 资源级授权
   - 是否允许上传
   - 是否允许查询某个任务
   - 是否允许播放某个视频
   - 是否允许修改元数据

#### 推荐落地方式

如果 Java 是主业务系统：

1. Java 负责业务鉴权与授权决策
2. `video-service` 校验 Java 签发的短期 token 或 service token

这样可以形成：

- Java 管控制面
- Go 管视频能力面

### 4.2 上传接口支持业务上下文

#### 当前问题

当前上传接口更像“上传文件并开始转码”，但作为下游服务，上游通常不仅关心文件本身，还关心：

1. 归属课程/章节/业务单据
2. 发起人是谁
3. 来源系统是谁
4. 是否需要回调
5. 请求链路标识是什么

#### 建议

在上传接口或上传会话接口中支持业务元数据，例如：

1. `biz_type`
2. `biz_id`
3. `source_system`
4. `operator_id`
5. `callback_url`
6. `trace_id`

即使第一版不全部落地，也建议预留字段结构。

### 4.3 异步任务状态机标准化

#### 当前问题

当前项目已经有转码状态，但如果作为长期下游服务，上游依赖的是稳定的任务状态契约，而不是内部实现细节。

#### 建议

明确标准化状态机，例如：

1. `uploaded`
2. `queued`
3. `processing`
4. `retrying`
5. `done`
6. `failed`
7. `dead`

并在查询接口中稳定返回：

1. `status`
2. `stage`
3. `retry_count`
4. `last_error_code`
5. `last_error_message`
6. `updated_at`

### 4.4 幂等能力

#### 当前问题

一旦上游系统、网关或用户行为触发重试，如果没有幂等机制，就容易出现：

1. 重复视频记录
2. 重复转码任务
3. 重复回调
4. 状态混乱

#### 建议

优先补两类幂等：

1. 上传幂等
   - 支持 `idempotency_key`
   - 或 `request_id`
2. 回调幂等
   - 支持重试
   - 支持去重

### 4.5 回调或事件通知能力

#### 当前问题

当前更偏向上游系统通过轮询查询任务状态。轮询可以保留，但不适合作为长期唯一模式。

#### 建议

补充至少一种异步通知机制：

1. HTTP callback / webhook
2. 后续可扩展事件消息推送

回调内容建议包含：

1. `task_id`
2. `video_id`
3. `status`
4. `play_url` / `hls_url`
5. `error_code`
6. `error_message`
7. `timestamp`

推荐模式：

1. 主通道：回调
2. 兜底：轮询

### 4.6 错误码体系平台化

#### 当前问题

虽然当前接口已经有统一 JSON 结构，但作为下游服务，还需要稳定、清晰、可分类的错误码体系。

#### 建议

至少区分以下类别：

1. 参数错误
2. 权限错误
3. 资源不存在
4. 状态冲突
5. 依赖异常
6. 系统繁忙
7. 异步任务失败
8. 可重试/不可重试异常

可参考错误码：

1. `invalid_argument`
2. `unauthorized`
3. `forbidden`
4. `video_not_found`
5. `task_not_found`
6. `task_conflict`
7. `rate_limited`
8. `storage_timeout`
9. `transcode_failed`
10. `vectorize_failed`
11. `service_unavailable`

## 5. P1：很建议补齐的能力

这些能力不一定是第一天必须上线，但很快会成为上游系统接入和治理的痛点。

### 5.1 控制面 / 数据面职责清晰分离

#### 建议

将接口按两类能力明确区分：

1. 控制面
   - 创建上传会话
   - 查询任务状态
   - 获取播放地址
   - 删除视频
   - 更新发布/推荐状态
   - 推荐查询
2. 数据面
   - 上传视频流
   - 访问 HLS / 封面 / 对象存储代理资源

这样可以帮助上游系统明确：

1. 元数据和权限逻辑走控制面
2. 大流量字节传输走数据面

### 5.2 标准上传体系

#### 当前问题

当前上传能力可用，但下游服务化后，应该抽象成标准上传模型，而不是单一接口。

#### 建议

形成一套完整上传能力：

1. 创建 upload session
2. 上传 part
3. 查询 session
4. 完成 session
5. 取消 session

这样可以同时适配：

1. 小文件直传
2. 大文件分片上传
3. 失败恢复

### 5.3 风暴保护和流量治理

#### 当前问题

项目已经在内部开始增强 worker 可靠性，但如果作为下游服务，仅靠 worker 内部补强是不够的。

#### 建议

在入口层和调度层同时增加：

1. 上传准入控制
2. 队列 backlog 保护
3. worker 总量闸门
4. 过载时标准化返回
5. 动态限流

例如：

1. 队列积压过大时只允许创建会话，不允许立即 complete
2. 过载时直接返回 `429` 或 `service busy`
3. 在任务状态里明确展示排队状态

### 5.4 面向服务治理的可观测性

#### 当前状态

项目已经有比较有价值的开发排障日志，例如：

1. vector task `total_cost_ms`
2. hierarchical segment 原始/归一化日志
3. retry / failed 日志

#### 建议补充

从服务治理视角增加指标和观测维度：

1. API 层
   - 请求量
   - 错误率
   - P95/P99
   - 来源系统维度
2. 任务层
   - 入队速率
   - dequeue 速率
   - 成功率
   - retry rate
   - dead-letter count
3. 依赖层
   - Redis 延迟
   - PostgreSQL 延迟
   - S3 timeout rate
   - ASR / LLM timeout rate
   - FFmpeg failure count

并保证可以按以下关键字段串联链路：

1. `task_id`
2. `video_id`
3. `source_system`
4. `trace_id`

### 5.5 正式接入文档

#### 当前状态

当前 README 已经具备较好的说明性，但对于多个上游系统长期接入来说，还需要正式接入文档。

#### 建议补充内容

1. 鉴权方式
2. 上传模式
3. 任务状态机定义
4. 错误码列表
5. 回调协议
6. 幂等约定
7. 限流规则
8. SLA / 超时建议
9. Java 接入示例

## 6. P2：中长期优化

这些能力更偏平台化和规模化治理，不一定需要立刻做，但后续演进时很重要。

### 6.1 拆分 combined worker

#### 当前状态

当前 `cmd/worker` 会同时运行：

1. transcode worker
2. vector worker

#### 建议

中长期拆成可独立扩缩容的运行单元：

1. transcode worker
2. vector worker

#### 原因

两类任务资源模型不同：

1. 转码更吃 CPU / 磁盘
2. 向量化更吃外部 AI / FFmpeg / I/O

拆开后更容易：

1. 单独扩缩容
2. 单独限流
3. 单独止损

### 6.2 统一并发预算控制

#### 当前问题

当前部分并发参数语义仍然偏实现细节，例如 `CoarseWorkers` 既影响视频级并发，也影响单视频内部并发。

#### 建议

未来将并发配置拆得更明确：

1. 视频级 worker 数
2. 单视频 coarse 并发数
3. refine ASR 并发数
4. 全局 FFmpeg 并发预算
5. 全局 AI 调用并发预算

### 6.3 直传对象存储

#### 建议

在文件更大、上游更多时，逐步演进到：

1. 服务端签发上传会话或上传凭证
2. 客户端直传对象存储
3. 完成后通知服务入库并触发后续转码

这会显著降低大文件上传对服务实例本身的压力。

### 6.4 API 版本化

#### 建议

长期建议统一版本化：

1. `/api/v1/...`

避免长期同时维护新旧风格接口，并为后续演进留出空间。

## 7. 推荐优先级路线

### 第一梯队：必须尽快做

1. 鉴权 / 授权标准化
2. 任务状态机标准化
3. 幂等机制
4. 回调 / webhook
5. 错误码体系

### 第二梯队：很建议做

1. 控制面 / 数据面职责分离
2. 标准上传体系
3. 风暴保护
4. 面向治理的可观测性
5. 正式接入文档

### 第三梯队：中长期演进

1. 拆分 combined worker
2. 统一并发预算控制
3. 直传对象存储
4. API 版本化

## 8. 如果当前只能先做 5 件事

如果当前资源有限，建议优先落地：

1. 统一鉴权模型
2. 定义稳定的任务状态机
3. 补幂等 key
4. 补回调 / webhook
5. 补错误码 + 风暴保护

完成这 5 项之后，`video-service/` 就会从“功能完整的视频后端”更进一步，变成“适合被上游系统稳定依赖的下游能力服务”。

## 9. 结论

当前 `video-service/` 已经具备较完整的视频能力和异步处理能力，但要真正作为多个上游系统的长期下游服务，最需要补的是服务治理能力，而不是继续堆单点业务功能。

一句话总结：

1. 现在它已经是一个可运行的视频能力服务
2. 但要成为成熟下游服务，优先级最高的是鉴权、幂等、任务治理、回调、错误码和风暴保护
