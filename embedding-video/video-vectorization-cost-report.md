## 7. 成本结构

按本次样本看，成本结构如下：

| 成本项 | 费用 | 占比判断 |
|---|---:|---|
| ASR | `约 1.253 元` | 绝对大头，约 `92%-97%` |

- 后续如果每小时音频总量变化不大，总体成本会比较稳定。

## 8. 改用本地部署 ASR 的隐患


> 这不是单纯的“把 API 地址换成本地地址”。

当前系统已经连续开发约 `2` 个月，ASR 已经深度嵌入现有视频向量化链路。此时切换到本地部署 ASR，隐患主要不在“能不能接上”，而在**会不会打乱当前已经逐步稳定的处理质量、性能和交付节奏**。

### 8.1 当前实现对现有 ASR 路径已有实际耦合

从现有代码看，ASR 并不是一个孤立插件，而是嵌入在以下多个关键阶段：

- 样本模式下的窗口 ASR
- hierarchical 模式下的粗分段 ASR
- refine 阶段的再次 ASR
- 边界对齐、尾句补齐过程中的二次/多次转写
- DashScope 兼容接口失败后的 WebSocket 回退链路

这意味着本地 ASR 一旦替换，不只是影响“文字识别是否成功”，还会影响后续：

- 分段边界判断
- 摘要重写输入
- embedding 文本内容
- 检索召回质量
- 推荐结果稳定性

换句话说，**ASR 输出文本本身就是当前整个向量化质量的上游输入**。只要文本风格、断句、标点、专有名词识别、口语噪声处理方式变化，后面的链路都会被放大影响。

### 8.2 最大风险不是接入失败，而是质量回退不容易第一时间看出来

本地 ASR 的直接风险通常包括：

- 识别准确率下降
- 断句风格变化
- 标点补全能力变化
- 数学、地理、人名、课程术语识别不稳定
- 长音频、重口音、噪声环境下效果波动

这些问题不一定会让任务报错，但会产生更隐蔽的后果：

- 视频仍然“处理成功”
- 向量也能正常入库
- 前端看起来流程正常
- 但检索命中率、摘要质量、推荐效果会变差

这种问题比直接报错更难发现，因为它往往是**业务效果缓慢变差**，而不是系统立即失败。

### 8.3 当前 refine 和边界对齐逻辑对 ASR 文本质量比较敏感

现有链路已经做了较多围绕 ASR 文本的二次处理，例如：

- boundary alignment
- tail alignment
- selective refine
- summary rewrite

这些逻辑默认前提是：ASR 文本在句尾、停顿、段落语义上有一定稳定性。

如果本地 ASR 的输出风格与当前云端模型不同，可能出现：

- 句子过早截断，导致边界对齐误判
- 句尾补齐失效，导致摘要输入不完整
- refine 后文本更短、更碎，embedding 语义密度下降
- 原本已调好的阈值和经验参数全部需要重新校准

这类问题不会只影响一个函数，而是会把近两个月里围绕当前 ASR 逐步打磨出来的行为基线打散。

### 8.4 并发、超时和资源模型会发生变化

当前 worker 的 ASR 并发、pool 上限、超时控制，都是围绕当前远端 ASR 服务形态设计的。

如果切到本地部署，资源瓶颈会从“外部 API 计费”转成“本机或本地集群算力”：

- CPU/GPU 是否足够
- 显存是否够支撑并发
- 模型 warm-up 是否稳定
- 单机吞吐是否会导致队列积压
- 多 worker 并发时是否会相互争抢资源
- 长音频是否触发本地推理超时或 OOM

这意味着改本地 ASR 后，需要重新验证：

- worker 并发配置
- 单任务超时
- Redis 队列积压情况
- 机器规格
- GPU 利用率与峰值
- 批量处理时的稳定性

如果不重做这轮验证，最容易出现的情况就是：

- 云端 API 成本下降了
- 但本地机器跑不满、跑不稳，整体吞吐反而下降

### 8.5 运维复杂度会明显上升

云端 ASR 的隐性价值之一，是把模型部署、扩缩容、升级兼容、运行异常处理转移给供应商。

换成本地部署后，团队需要自己承担：

- 模型服务部署
- GPU 驱动和推理框架兼容
- 服务重启与健康检查
- 负载均衡
- 版本升级与回滚
- 监控告警
- 容量规划

对于一个已经开发了 `2` 个月、当前主目标本应逐步收敛业务效果的项目来说，这会把团队注意力从“提升检索与推荐质量”转移到“维护推理基础设施”上。

### 8.6 切换时间点本身就是风险

“是否使用本地 ASR”这个决策，如果放在项目一开始做，主要是架构选型问题。

但现在项目已经开发了约 `2` 个月，再切换，风险会变成**阶段性返工**：

- 需要重新跑已有样本集做质量对比
- 需要重新确认成本模型，不只是 API 成本，还包括机器成本
- 需要重新做压测
- 需要重新评估检索质量
- 需要重新确认线上异常处理策略
- 可能推迟当前业务交付时间

此时最怕的是“为了省后续调用费，提前引入了新的不确定性”，结果导致：

- 成本没马上省多少
- 交付节点先被打乱
- 质量又要重新回归验证

### 8.7 本地部署不等于一定更便宜

从当前测算看，现阶段模型成本的大头确实是 ASR。

但本地部署后，成本并不会自动变成接近零，因为还要考虑：

- GPU 机器成本
- 机器空载和峰值冗余
- 运维成本
- 故障排查成本
- 模型升级与验证成本

在处理规模还没有大到足以稳定摊薄固定成本之前，“本地 ASR 更便宜”常常只是表面结论。

尤其是在当前阶段，项目还处于质量和链路稳定性优先的阶段，**过早把外部 API 成本问题转成内部工程复杂度问题，未必划算**。

### 8.8 更稳妥的建议

如果确实要评估本地部署 ASR，建议不要直接替换现网主链路，而应该采用：

- 并行验证
- 小样本 A/B 对比
- 固定语料回归评测
- 灰度切流

更稳妥的做法是：

1. 先保留当前云端 ASR 作为主链路。
2. 单独接一个本地 ASR 适配层。
3. 用同一批视频对比：
   - 识别准确率
   - 分段稳定性
   - embedding 后检索效果
   - 单小时真实处理成本
   - 吞吐和失败率
4. 只有在“质量不退、吞吐可接受、总体成本确实下降”三者同时成立时，再考虑切主链路。

### 8.9 结论

对于当前这个已经开发了约 `2` 个月的系统，本地部署 ASR 的主要隐患不是“接不进去”，而是：

- **会放大已开发链路中的耦合点**
- **会引入新的质量回归风险**
- **会打乱当前已经逐步形成的参数和行为基线**
- **会把成本问题转成基础设施和交付风险问题**

因此，更合理的判断应当是：

> 本地 ASR 可以评估，但不适合在当前阶段直接替换现有主链路，更不适合在没有完整 A/B 验证和压测结论前仓促切换。

## 9. 充值额度建议

如果目标是稳定处理 `1000` 小时视频素材，不建议只充值理论最低值，因为实际运行中还会有以下偏差：

- 某些视频会触发更多重试
- 部分音频可能因为切段策略导致 ASR 总秒数略高于素材时长
- `qwen-plus` 真实计费口径可能落在更高档位
- 后续模型价格可能调整
- 账单统计和实际业务波峰需要留出缓冲

建议分三档准备：

| 建议档位 | 金额 | 说明 |
|---|---:|---|
| 最低可用 | `1500 元` | 覆盖 `1000` 小时主体成本，并留出约 `16%-22%` 缓冲 |
| 稳妥推荐 | `1800 元` | 更适合生产跑批，能覆盖重试、波动和少量额外调用 |
| 宽松推荐 | `2000 元` | 适合希望减少中途补充值、保留明显安全边际的场景 |

我的建议是：

- **如果只是先验证一轮 1000 小时处理能力，充 `1500 元` 可以开工。**
- **如果这是正式生产批量处理，建议直接准备 `1800-2000 元`。**

## 10. 最终建议

如果你要做 `1000` 小时规模的正式处理，建议按下面口径决策：

1. 预算测算按 **`1300 元左右`** 作为纯模型理论成本。
2. 财务或运营准备按 **`1800 元`** 作为稳妥充值额度。
3. 如果不希望中途因为波动补充值，直接准备 **`2000 元`** 更省事。

一句话总结：

> 当前这套视频向量化链路，`1000` 小时素材的模型成本大约在 **1230-1290 元**，建议实际充值 **1500 元起步，1800-2000 元更稳妥**。
